# 📡 Quicksave Offline-First Sync Engine

**Last Updated:** June 30, 2026  
**Status:** Production Ready  

## 📖 Overview
The Quicksave app is designed for users in areas with highly volatile network connectivity (e.g., commuting in Lagos). To ensure users never lose financial data or face blocked UI states due to network drops, we implemented a robust Offline-First Sync Engine.

This engine allows users to queue actions (like making a Contribution) locally while offline, and automatically flushes the queue to the backend once a stable connection is restored.

---

## 🏗️ Architecture Flow

### 1. Offline Detection (`networkSlice.ts` & `OfflineBanner.tsx`)
We use `@react-native-community/netinfo` to actively monitor connectivity. 
- If `isConnected` or `isInternetReachable` becomes `false`, the Redux `networkSlice` updates.
- A global `<OfflineBanner />` slides down to inform the user.
- Dangerous actions (like Bank Withdrawals) are visually disabled in the UI.

### 2. Local Storage (`offlineQueue.ts` -> SQLite)
We use **Expo SQLite** instead of `AsyncStorage` for true ACID-compliant persistence.
- When an offline user submits a contribution, we generate a unique `uuid` and save the payload to SQLite with a `PENDING` status.
- The Redux `offlineQueueSlice` reads this database and updates the UI to show a `<PendingSyncBadge />` on the group card.

### 3. The Sync Trigger (`syncOfflineData` Thunk)
The moment `NetInfo` detects internet restoration:
- Redux automatically dispatches `syncOfflineData()`.
- The app pulls all `PENDING` records from SQLite and sends them via `POST /api/v1/groups/sync-contributions`.
- **UI State:** Tracks `isSyncing` and `syncAttempts` to show loading spinners or error states if the server rejects the sync.

### 4. Backend Processing (`sync.queue.ts`)
The Express backend receives the array of offline contributions and immediately pushes them to a **Bull Queue** (Redis) and returns a `202 Accepted` to the mobile app.
- **Why?** We don't block the HTTP request doing heavy database math. The mobile app deletes its SQLite records immediately upon the `202` response.

### 5. Idempotency & Conflict Resolution
What if the mobile app sends the sync payload twice because of a network timeout?
- The Bull worker extracts the unique `offlineId` generated by the mobile app's SQLite DB.
- It queries the `Transaction` ledger: `reference: SYNC_${offlineId}_DEBIT`.
- If the transaction exists, the worker safely skips the job (preventing double-charging).
- If it doesn't exist, it processes the massive atomic `$transaction` to deduct the user wallet and credit the group vault.

### 6. Background Retry Strategy
If the backend fails to process the job (e.g., database lock or sudden insufficient funds):
- Bull Queue utilizes **Exponential Backoff**.
- It will automatically retry the job 5 times, waiting `2s, 4s, 8s, 16s` between attempts before marking it as permanently failed.

### 7. Real-Time UI Hydration
Upon successful backend processing, the worker triggers `groupService.logAndBroadcast()`. 
- Socket.io emits a `newActivity` event.
- The mobile app (which is now online) instantly receives this event and updates the user's dashboard balance and group progress bar without requiring a pull-to-refresh.
